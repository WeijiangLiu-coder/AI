# LangChain 核心框架介绍

`LangChain` 是一个用于构建大模型应用的开发框架。它的目标不是替代大模型，而是把“大模型调用、Prompt 组织、知识检索、工具调用、工作流编排”这些零散能力串起来，帮助开发者更稳定地构建 AI 应用。

---

## 1. LangChain 是什么

可以把 `LangChain` 理解为一层“大模型应用开发框架”。

如果只直接调模型 API，你通常只能完成：
- 发一个 Prompt
- 拿回一个回答

但真实业务里往往还需要：
- 组织多段 Prompt
- 管理上下文
- 检索知识库
- 调用外部工具
- 把多个步骤串成工作流
- 统一输出格式

`LangChain` 解决的就是这些“工程化问题”。

---

## 2. LangChain 解决了什么问题

### 2.1 解决“只会调模型，不会做应用”的问题

直接调用大模型 API 只能得到一次回答，但业务场景通常需要：
- 多轮对话
- 固定输出格式
- 外部知识补充
- 工具调用
- 多步骤处理

`LangChain` 把这些能力封装成统一的组件，方便组合使用。

### 2.2 解决 Prompt、模型、工具彼此割裂的问题

没有框架时，代码里常见问题是：
- Prompt 写在各处，难维护
- 模型调用逻辑分散
- 工具调用接口不统一
- 检索和生成流程混在一起

`LangChain` 提供统一抽象，让这些部分能按模块拆开。

### 2.3 解决大模型“不知道外部信息”的问题

大模型本身不知道你的私有知识库、数据库和实时接口数据。

`LangChain` 可以把：
- 向量检索
- 文档加载
- 数据库查询
- API 调用

接进来，形成 `RAG` 或 Tool Calling 流程。

### 2.4 解决复杂任务难编排的问题

例如一个任务可能要经过：
1. 读取文档
2. 检索相关内容
3. 调用模型总结
4. 校验格式
5. 写入结果

`LangChain` 可以把这些步骤串成链路或工作流，而不是全写成零散代码。

---

## 3. LangChain 的核心定位

一句话概括：

**LangChain 是帮助你把大模型能力工程化、模块化、可组合化的框架。**

它适合做的不是“训练模型”，而是“搭应用”。

所以它更关注：
- 应用层开发
- 工作流编排
- 检索增强
- 工具调用
- 结构化输出

而不是底层训练本身。

---

## 4. LangChain 的核心架构

如果只说“LangChain 有很多组件”，会比较散。更容易理解的方法是把它看成一条从“用户输入”到“结果输出”的应用架构链路。

它的核心架构可以概括成 5 层：

1. **输入组织层**
- 接收用户问题、系统设定、上下文变量
- 通过 `Prompt Templates` 把静态模板和动态数据拼起来

2. **模型交互层**
- 通过 `Model I/O` 与不同大模型交互
- 负责把输入送进模型，再把输出取回来

3. **能力增强层**
- 通过 `Retrievers` 接知识库、向量数据库，实现 `RAG`
- 通过 `Tools` 接 API、数据库、代码执行器等外部能力
- 通过 `Memory` 管理多轮上下文

4. **流程编排层**
- 通过 `Chains` 把固定步骤串起来
- 通过 `Agents` 在复杂任务里做动态决策，决定下一步调什么工具、走什么流程

5. **结果处理层**
- 通过 `Output Parsers` 对模型结果做结构化解析、格式校验和下游适配

一句话看这条链路：

**用户输入 -> Prompt 组织 -> 模型调用 -> 检索/工具/记忆增强 -> 链或 Agent 编排 -> 输出解析 -> 返回结果**

如果画成最简流程，可以这样理解：

```text
用户问题
  -> Prompt Templates
  -> Model I/O
  -> (Retrievers / Tools / Memory)
  -> Chains 或 Agents
  -> Output Parsers
  -> 最终结果
```

这就是 LangChain 的核心架构本质：

- 不是单一模型
- 不是单一 Prompt
- 而是一套“围绕大模型的应用执行框架”

所以你在学 LangChain 时，不能只把它看成“Prompt 工具库”，而要把它看成：

**大模型应用的模块化分层架构。**

---

## 5. LangChain 的核心组件

不同版本命名会有一些变化，但核心思想基本稳定，可以先抓住这几个部分。

### 5.1 Model I/O

负责和模型交互，包括：
- Prompt 模板
- 聊天模型调用
- 输出解析

你可以把它理解成“输入模型之前”和“拿到结果之后”的这一层。

典型作用：
- 统一写 Prompt
- 替换不同模型供应商
- 把输出转成结构化结果

### 5.2 Prompt Templates

用于把固定模板和动态变量拼接起来。

例如：
- 角色设定
- 用户问题
- 知识库片段
- 输出格式要求

都可以用模板方式统一管理。

这样做比把 Prompt 写死在字符串里更清晰，也更利于复用。

### 5.3 Output Parsers

用于约束和解析模型输出。

因为模型天然容易：
- 输出格式不稳定
- 多说废话
- JSON 不合法

所以需要解析器帮助你把结果转成：
- JSON
- 列表
- 字段对象

这对“自动化处理”和“接下游系统”很重要。

### 5.4 Retrievers

检索器负责“先查资料，再回答”。

它通常会和这些能力搭配：
- Embedding
- 向量数据库
- 文档切分
- 相似度搜索

这是做 `RAG` 的关键一层。

### 5.5 Chains

`Chain` 指把多个步骤串成一个调用链。

例如：
1. 先格式化 Prompt
2. 再调用模型
3. 再解析输出

这就是一个最基本的链。

它解决的是“多步骤流程复用”的问题。

### 5.6 Tools

`Tool` 是给模型提供外挂能力。

例如：
- 查天气
- 查数据库
- 调接口
- 执行代码

模型本身不会真的直接调用你的系统，而是通过工具协议让程序去执行。

### 5.7 Agents

`Agent` 是让模型基于目标自主决定：
- 先做什么
- 要不要调用工具
- 下一步怎么办

它和普通链的区别是：
- `Chain` 更固定
- `Agent` 更灵活

但灵活也意味着更复杂、更难控。

### 5.8 Memory

`Memory` 用于管理多轮对话上下文。

例如：
- 记录前面对话
- 保留用户偏好
- 让模型知道“我们刚刚聊过什么”

它解决的是“多轮对话连贯性”的问题。

---

## 6. LangChain 的典型工作方式

一个典型 AI 应用流程可能是：

1. 用户提问
2. Prompt 模板整理输入
3. 检索器去知识库找相关内容
4. 把问题和检索结果一起交给模型
5. 模型生成答案
6. 输出解析器把结果转成规定格式

如果任务更复杂，还可能加入：
- Tool 调用
- Agent 决策
- Memory 记忆

---

## 7. LangChain 和 RAG、Agent 的关系

### LangChain 和 RAG

`RAG` 是一种应用方案，核心是：
- 先检索
- 再生成

`LangChain` 不是 RAG 本身，而是一个很适合实现 RAG 的框架。

也就是说：
- `RAG` 是方法
- `LangChain` 是实现方法的工具框架

### LangChain 和 Agent

`Agent` 也不是 LangChain 独有的概念，但 LangChain 提供了实现 Agent 的基础组件：
- 模型
- 工具
- 状态流转
- 任务编排

所以可以说：
- `LangChain` 可以做 Agent
- 但 `LangChain` 不等于 Agent

---

## 8. LangChain 和 LangGraph 的区别

很多人学到 `Agent` 这里时，会继续问一个问题：

**既然 LangChain 也能做 Agent，为什么又会有 LangGraph？**

核心答案是：

- `LangChain` 可以做 Agent
- `LangGraph` 更适合做复杂 Agent

### 8.1 抽象层级不同

- `LangChain` 更像一套大模型应用组件框架
- `LangGraph` 更像一套面向 Agent 的工作流编排框架

`LangChain` 重点在于把：
- Prompt
- Model
- Retrieval
- Tools
- Output Parser

这些能力接起来。

而 `LangGraph` 更强调：
- 节点
- 边
- 状态
- 分支
- 循环
- 恢复

也就是把 Agent 执行过程当成一张“状态图”来管理。

### 8.2 适合的流程复杂度不同

如果任务比较简单，例如：
- 问题进来
- 查一次知识库
- 调一次工具
- 输出结果

那 `LangChain` 通常就够用了。

如果任务比较复杂，例如：
- 先查日志
- 再判断是否查数据库
- 信息不够就继续追问用户
- 某一步失败要重试
- 高风险步骤要人工确认

这种就更适合 `LangGraph`。

### 8.3 对 Agent 的控制力不同

`LangChain` 能做 Agent，但在复杂场景下更像“拼出一个 Agent”。

`LangGraph` 则更像“严肃地管理一个 Agent 的状态流转和执行过程”。

所以：
- 简单 Agent：`LangChain` 可以胜任
- 复杂 Agent：`LangGraph` 更有优势

### 8.4 是否支持状态持久化和恢复

复杂 Agent 场景经常会遇到这些需求：
- 执行到一半中断
- 需要从上一步继续
- 需要保留中间状态
- 需要人工介入后再继续跑

`LangGraph` 对这类“状态管理”支持更强，这也是它适合生产级 Agent 的重要原因。

### 8.5 一句话理解

- `LangChain`：更适合“把能力接起来”
- `LangGraph`：更适合“把复杂流程管起来”

### 8.6 举个例子

#### 适合 `LangChain` 的场景

做一个知识库问答助手：
- 用户提问
- 检索知识库
- 拼 Prompt
- 模型回答

这是典型的线性流程，`LangChain` 很合适。

#### 适合 `LangGraph` 的场景

做一个自动问题排查 Agent：
- 读取用户报错
- 判断是否查日志
- 查完日志后判断是否查数据库
- 如果信息不够就继续追问
- 如果要高风险操作就等人工审批
- 最后生成处理建议

这种有分支、有循环、有状态的任务，更适合 `LangGraph`。

---

## 9. 为什么 LangChain 会流行

因为它降低了大模型应用开发门槛。

在没有框架时，开发者要自己处理很多基础问题：
- Prompt 如何组织
- 数据如何检索
- 工具如何调用
- 流程如何串联
- 输出如何规范化

`LangChain` 把这些常见能力抽象出来，让你能更快从“会调模型”走到“能做应用”。

---

## 10. LangChain 的优点

- 组件化强，适合快速搭建原型
- 对 `RAG`、`Tool`、`Agent` 支持丰富
- 生态资料多，学习资源多
- 适合把大模型能力接进业务系统

---

## 11. LangChain 的局限

- 抽象层较多，新手一开始容易觉得概念多
- 版本迭代较快，不同教程代码可能不完全一样
- 复杂 Agent 场景下，调试成本不低
- 小项目如果只做一次简单调用，直接写 API 可能更轻量

所以它不是“所有场景都必须用”，而是适合：
- 有检索需求
- 有工具调用需求
- 有多步骤流程需求
- 想把 AI 应用做得更工程化

---

## 12. 一个简单例子

假设你要做一个“公司知识库问答助手”。

不用 `LangChain` 时，你要自己处理：
- 文档切分
- 向量化
- 检索
- 拼 Prompt
- 调模型
- 整理输出

用了 `LangChain` 后，可以拆成更清晰的模块：
- 文档加载器加载知识库
- 检索器查找相关片段
- Prompt 模板组织提问和上下文
- 聊天模型生成答案
- 输出解析器整理结果

这样整个系统会更容易维护和扩展。

---

## 13. 学 LangChain 时最该抓住什么

初学时不要一上来就记很多类名，先抓住 4 个核心问题：

1. **怎么把 Prompt 管好**
2. **怎么把外部知识接进来**
3. **怎么把工具接进来**
4. **怎么把多步流程串起来**

如果这 4 个问题理解了，LangChain 的主体框架就算抓住了。

---

## 14. 一句话总结

**LangChain 不是训练模型的框架，而是构建大模型应用的框架。它主要解决 Prompt 管理、知识检索、工具调用、流程编排和输出规范化这些工程问题。**
