# 大模型核心术语手册

这份手册只收录第 1 周真正用到、且彼此容易混淆的核心概念。每个术语都按统一模板整理，方便复习、串讲和后续继续补充。

---

## 1. `temperature`

### 是什么
用于调节模型输出时的随机性。数值越低，回复越稳定；数值越高，回复越发散。

### 解决什么问题
控制回复在“准确稳定”和“灵活有创意”之间的平衡。

### 和相近概念的区别
- 和 `top_p`、`top_k` 一样都影响采样，但 `temperature` 是对整体概率分布做全局调节。
- `temperature` 偏向调“发散程度”，`top_p` / `top_k` 偏向调“候选范围”。

### 典型使用场景
- 事实问答、代码生成：`0.0 ~ 0.3`
- 通用对话：`0.4 ~ 0.7`
- 创意写作：`0.8+`

### 一句话例子
写代码时把 `temperature` 设低一点，模型更不容易天马行空。

---

## 2. `top_p`

### 是什么
核采样参数。只在累计概率达到 `p` 的那批候选词里采样。

### 解决什么问题
动态收缩候选词范围，让模型既保留一定随机性，又不至于过于发散。

### 和相近概念的区别
- 和 `top_k` 相比，`top_p` 不是固定保留多少个词，而是按累计概率截断。
- 和 `temperature` 相比，`top_p` 不改变分布形状，主要决定“可选池子有多大”。

### 典型使用场景
- 通用聊天、问答、内容生成
- 通常和 `temperature` 搭配使用

### 一句话例子
`top_p = 0.9` 表示只在最可能、累计概率前 90% 的候选里挑词。

---

## 3. `top_k`

### 是什么
每次生成时只保留概率最高的前 `k` 个候选词，再从这 `k` 个里采样。

### 解决什么问题
用一个固定数量来限制随机候选集，减少跑偏。

### 和相近概念的区别
- `top_k` 是固定数量截断，`top_p` 是动态概率截断。
- `top_k = 1` 时几乎接近贪心输出，稳定但容易机械。

### 典型使用场景
- 某些支持 `top_k` 的推理框架
- 需要更细粒度约束候选词数量时

### 一句话例子
`top_k = 40` 可以理解成“每一步只看前 40 个最可能的词”。

---

## 4. 幻觉（Hallucination）

### 是什么
模型生成了读起来很像真的、但事实上没有依据或者明显错误的内容。

### 解决什么问题
它本身不是“用来解决问题”的能力，而是一个必须管理的风险点。

### 和相近概念的区别
- 幻觉不等于普通答错，它更强调“看起来很合理、实际上无根据”。
- 幻觉也不完全由采样参数导致，知识缺失、上下文不足、任务设计不当都会触发。

### 典型使用场景
- 事实问答中编造出处
- 代码生成中编造函数名
- 实时信息场景里凭空给出数据

### 一句话例子
模型一本正经地给出一篇并不存在的论文标题，就是典型幻觉。

---

## 5. Embedding（嵌入）

### 是什么
把文本、图片等内容转换成一串向量，用数字表示其语义特征。

### 解决什么问题
让机器能基于“语义相似度”而不是“字面是否完全相同”来做检索、聚类和推荐。

### 和相近概念的区别
- Embedding 模型主要负责“表示”，聊天模型主要负责“生成”。
- Embedding 不直接回答问题，通常给 RAG、向量数据库、推荐系统提供输入。

### 典型使用场景
- 语义搜索
- 知识库检索
- 文本去重、聚类、召回

### 一句话例子
“退款流程”和“怎么退钱”字面不同，但 Embedding 后可能会很接近。

---

## 6. 向量数据库（Vector Database）

### 是什么
专门存储向量并支持相似度检索的数据库，常会一起保存文档 ID、标题、来源等元数据。

### 解决什么问题
高效找到“和当前问题最相近”的知识片段，为 RAG 提供检索结果。

### 和相近概念的区别
- 普通数据库更擅长精确查询，向量数据库更擅长语义相似查询。
- Embedding 是把内容变成向量，向量数据库是把这些向量存起来并检索。

### 典型使用场景
- 企业知识库
- 智能客服
- 文档问答

### 一句话例子
用户问“如何开发票”，系统先把问题转成向量，再去向量库里找最相关文档段落。

---

## 7. Prompt

### 是什么
你发给模型的输入内容，包括问题、指令、背景资料、格式要求等。

### 解决什么问题
告诉模型“这次要做什么、按什么方式做、基于哪些材料做”。

### 和相近概念的区别
- Prompt 是广义输入；`System Prompt` 是其中更高优先级的一层约束。
- Prompt 解决的是“当前这次任务怎么做”，不是永久改模型行为。

### 典型使用场景
- 提问
- 让模型总结、翻译、写代码
- 要求按固定格式返回

### 一句话例子
“请基于以下会议纪要，输出 3 条行动项”就是一个 Prompt。

---

## 8. System Prompt

### 是什么
系统级提示词，用来规定模型的角色、边界、风格和全局规则，优先级通常高于普通用户输入。

### 解决什么问题
让模型在整个对话过程中保持统一行为，例如身份设定、输出语言、禁止事项和格式要求。

### 和相近概念的区别
- `Prompt` 更像本轮任务说明；`System Prompt` 更像全局规则。
- 它比临时提醒更稳定，但仍属于运行时约束，不等于微调。

### 典型使用场景
- 设定“你是代码助手”
- 规定“始终用中文回答”
- 要求“不确定时必须明确说不知道”

### 一句话例子
“你是一名资深 Python 工程师，回答必须简洁且先给结论”就是典型 System Prompt。

---

## 9. RAG（Retrieval-Augmented Generation）

### 是什么
先检索外部知识，再把检索结果连同用户问题一起交给大模型生成答案的方案。

### 解决什么问题
补足模型参数里没有、过期或不稳定的知识，降低幻觉，提高答案和私有知识库的一致性。

### 和相近概念的区别
- RAG 的重点是“先查再答”，主要解决知识补充问题。
- Agent 可能会用到 RAG，但 Agent 不只做检索。

### 典型使用场景
- 企业文档问答
- 客服知识库
- 产品手册助手

### 一句话例子
用户问报销流程，系统先去知识库里找制度文档，再让模型按文档内容回答。

---

## 10. Function Call（函数调用 / 工具调用）

### 是什么
模型按约定输出一个结构化调用请求，由程序去执行真实函数、API 或数据库查询，再把结果回传给模型。

### 解决什么问题
让模型获得“查实时数据、调用系统能力、执行动作”的能力，而不是只靠语言生成。

### 和相近概念的区别
- 它不是模型真的在代码环境里随意跑函数，而是“模型提出调用请求，程序代为执行”。
- 和 RAG 相比，RAG 更偏检索知识，Function Call 更偏触发工具或动作。

### 典型使用场景
- 查天气、查库存、查订单
- 调数据库
- 调用计算器、代码执行器

### 一句话例子
模型发现用户在问实时天气，于是请求调用 `get_weather(city="Hangzhou")`。

---

## 11. Agent

### 是什么
一种让模型具备“多步规划、调用工具、观察结果、继续决策”的应用形态。

### 解决什么问题
把复杂任务拆成多步执行，而不是只做一次性问答。

### 和相近概念的区别
- RAG 主要是检索知识，Agent 是基于目标做多步决策。
- Agent 常常会用到 RAG、Function Call、Memory 等能力，但不等于其中任何一个。

### 典型使用场景
- 自动查资料并生成报告
- 读取文件后写代码
- 多轮调用 API 完成任务

### 一句话例子
“先读需求文档，再查接口定义，再生成测试点”就是一个典型 Agent 工作流。

---

## 12. SFT（Supervised Fine-Tuning，监督微调）

### 是什么
用带答案的样本数据对基础模型继续训练，让模型更稳定地学会某种风格、格式或任务模式。

### 解决什么问题
把“长期稳定的行为倾向”写进模型，而不是每次都靠很长的 Prompt 提醒。

### 和相近概念的区别
- Prompt / Rule 是运行时提醒，SFT 是训练层面的长期改造。
- Skill / Tool 是外挂能力，SFT 是直接改变模型默认表现。

### 典型使用场景
- 客服风格统一
- 特定领域问答
- 固定格式抽取、分类、改写

### 一句话例子
如果希望模型长期像专业客服一样先道歉、再复述问题、再给方案，SFT 比一次次提醒更稳定。

## 13. 容易混淆的概念对

### `temperature` vs `top_p` vs `top_k`
- 三者都影响采样随机性。
- `temperature` 调整体发散程度。
- `top_p` 调动态候选范围。
- `top_k` 调固定候选数量。

### `Prompt` vs `System Prompt`
- `Prompt` 是这次任务的输入说明。
- `System Prompt` 是全局规则和角色设定。

### `Embedding` vs 向量数据库
- Embedding 负责把内容转成向量。
- 向量数据库负责存向量、查相似内容。

### `RAG` vs `Agent`
- RAG 强调“查资料后回答”。
- Agent 强调“多步决策并调用工具”。

### `Skill/Rule` vs `SFT`
- Skill / Rule 更像外挂流程和工程控制。
- SFT 更像把行为习惯训练进模型。

---

## 14. 一页速记

- 要更稳：降低 `temperature`，减小 `top_p` / `top_k`。
- 需要外部知识：优先考虑 `RAG`。
- 需要真实动作或实时数据：优先考虑 `Function Call` / Tool。
- 需要多步完成任务：考虑 `Agent`。
- 需要长期稳定的风格或行为：考虑 `SFT`。
